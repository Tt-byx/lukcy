# 📝 项目经历总结 - 毕业典礼抽奖系统（更新版）

---

## 一、简历版本（精简版）

### 项目名称
**毕业典礼实时互动抽奖系统**

### 项目描述
为大学毕业典礼设计并开发的高并发实时互动系统，支持 2000+ 用户同时在线参与弹幕互动和实时抽奖。采用前后端分离架构，包含大屏展示端、移动端和管理后台三端应用。使用 RabbitMQ 实现弹幕异步广播，Redisson 分布式锁保证抽奖并发安全，Sentinel 接口限流保护系统稳定性，JWT 无状态认证保障管理后台安全。

### 技术栈
**后端**：Spring Boot 3.2.5 + MyBatis-Plus + Spring Security + JWT + WebSocket + RabbitMQ + Redis + Redisson + Sentinel + MySQL 8.0

**前端**：Vue 3 + Vite + Composition API

**部署**：Docker + Docker Compose + Nginx + 阿里云 ECS

### 负责模块
1. **实时通信模块**：WebSocket 实现弹幕推送、抽奖状态同步、在线人数统计
2. **消息队列模块**：RabbitMQ 异步处理弹幕广播，业务逻辑与推送解耦
3. **分布式锁模块**：Redisson 控制抽奖并发，防止重复中奖
4. **安全认证体系**：Spring Security + JWT 实现管理后台无状态认证
5. **接口限流模块**：Sentinel 实现接口 QPS 控制，防止恶意请求
6. **三端前端开发**：大屏展示端（弹幕动画、抽奖滚动）、移动端（扫码注册、弹幕发送）、管理后台（活动管理、用户管理）
7. **容器化部署**：Docker 镜像构建、Docker Compose 编排、Nginx 反向代理

### 项目亮点
- RabbitMQ 消息队列实现弹幕异步广播，支持 2000+ 用户同时在线
- Redisson 分布式锁保证抽奖并发安全，锁粒度为轮次级别
- Sentinel 接口限流，弹幕 100 QPS、登录 10 QPS、抽奖 5 QPS
- Redis 缓存敏感词库，弹幕发送响应时间从 50ms 降至 < 5ms
- JWT 无状态认证，支持分布式部署
- Docker 一键部署，包含 MySQL、Redis、RabbitMQ、后端、前端五个容器

## 二、STAR 法则版本

### Situation（背景）
大学毕业典礼需要一个实时互动系统，支持 2000+ 名毕业生同时参与弹幕互动和在线抽奖。系统需要在大屏幕上实时展示弹幕和抽奖动画，学生通过手机扫码参与，管理员通过后台控制活动流程。

### Task（任务）
作为独立开发者，负责整个系统的设计、开发和部署：
1. 设计后端架构，实现活动管理、弹幕系统、抽奖系统等核心功能
2. 解决高并发场景下的性能和一致性问题
3. 实现安全的管理后台认证体系和接口限流
4. 开发三端前端应用，实现不同场景的用户交互
5. 完成容器化部署，确保生产环境稳定运行

### Action（行动）

**1. 架构设计**
- 采用前后端分离架构，后端 Spring Boot + 前端 Vue 3
- 设计 RESTful API，使用 Swagger 自动生成接口文档
- 分层架构：Controller → Service → Mapper，职责清晰
- DTO/VO 转换层，Entity 不直接暴露给前端

**2. 实时通信 + 消息队列**
- 使用 WebSocket 实现全双工通信
- RabbitMQ 消息队列实现弹幕异步广播
- 设计消息类型：弹幕、抽奖开始、抽奖结果、在线人数
- 业务逻辑和 WebSocket 推送解耦，支持水平扩展

**3. 高并发优化**
- **Redisson 分布式锁**：控制抽奖并发，锁粒度为轮次级别
  - 锁 key：`lucky:lottery:lock:{roundId}`
  - 等待时间 3 秒，持有时间 30 秒，支持自动续期
- **Redis 缓存**：敏感词库缓存，响应时间从 50ms 降至 < 5ms
- **Sentinel 限流**：接口级别 QPS 控制
  - 弹幕接口：100 QPS（高频）
  - 登录接口：10 QPS（防暴力破解）
  - 抽奖接口：5 QPS（核心资源）
- **乐观锁**：数据库 @Version 防止并发更新冲突

**4. 安全认证**
- Spring Security + JWT 实现无状态认证
- 公开接口（活动查询、弹幕发送）无需认证
- 管理接口（活动管理、用户管理）需要 JWT Token
- BCrypt 密码加密存储
- Sentinel 限流防止恶意请求

**5. 前端开发**
- 大屏端：Canvas 绘制二维码、CSS 动画实现弹幕滚动、抽奖滚动动画
- 移动端：响应式设计、学号快速登录、WebSocket 实时接收弹幕
- 管理端：活动 CRUD、抽奖轮次管理、用户禁言/移出、敏感词管理

**6. 部署优化**
- Docker 多阶段构建，减小镜像体积
- Nginx 配置 Gzip 压缩，减少传输体积 60%-70%
- 静态资源 7 天缓存，减少重复请求
- Docker Compose 编排五个服务：MySQL、Redis、RabbitMQ、后端、前端

### Result（结果）
1. **性能指标**
   - 弹幕延迟 < 100ms，支持 2000+ 用户同时在线
   - 敏感词检查响应时间从 50ms 降至 < 5ms（Redis 缓存）
   - 页面加载时间 < 2s（Gzip 压缩 + 静态资源缓存）
   - 接口限流保护，系统稳定性提升

2. **功能完成度**
   - 完整的三端应用：大屏端、移动端、管理后台
   - 核心功能：实时弹幕、滚动抽奖、敏感词过滤、用户管理
   - 安全认证：JWT 无状态认证、权限控制、接口限流
   - 高并发：消息队列、分布式锁、缓存优化

3. **部署效果**
   - Docker 一键部署，包含 5 个容器
   - 阿里云 ECS 生产环境稳定运行
   - 支持局域网和公网访问

## 四、面试话术

### Q: 如何保证 2000 人同时发弹幕不卡顿？
> A: 三层优化：
> 1. **消息队列**：弹幕发送后写入 RabbitMQ，立即返回，消费者异步广播
> 2. **Redis 缓存**：敏感词库缓存到 Redis，避免每次查数据库
> 3. **Sentinel 限流**：弹幕接口 100 QPS，超过直接返回 429

### Q: 如何防止抽奖重复中奖？
> A: 三层保护：
> 1. **Redisson 分布式锁**：锁粒度为轮次级别，同一轮次只能一个线程执行
> 2. **数据库乐观锁**：@Version 防止并发更新
> 3. **状态检查**：抽奖前检查轮次状态和参与者状态

### Q: 为什么选择 JWT 而不是 Session？
> A: 
> 1. JWT 无状态，支持分布式部署
> 2. 移动端友好，不需要 Cookie
> 3. 减少服务端存储压力
> 4. 可以携带用户信息（角色等）

### Q: RabbitMQ 解决了什么问题？
> A:
> 1. **解耦**：弹幕发送和 WebSocket 推送分离
> 2. **削峰**：2000 人同时发弹幕时，写入队列，消费者按能力消费
> 3. **异步**：业务线程不阻塞，立即返回
> 4. **扩展**：支持多消费者水平扩展

### Q: Sentinel 限流策略是什么？
> A: 按接口类型设置不同 QPS：
> - 弹幕接口：100 QPS（高频，预热 10 秒）
> - 登录接口：10 QPS（防暴力破解）
> - 抽奖接口：5 QPS（核心资源保护）
> - 全局默认：500 QPS
> 
> 采用预热模式，避免冷启动时突发流量打垮系统。

### Q: Docker 部署有什么优势？
> A:
> 1. 环境一致性：开发、测试、生产环境一致
> 2. 一键部署：docker-compose up -d 启动所有服务
> 3. 易于扩展：可以轻松增加服务实例
> 4. 便于迁移：镜像可以迁移到任何服务器
