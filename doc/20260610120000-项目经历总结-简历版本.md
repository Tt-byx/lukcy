# 📝 项目经历总结 - 毕业典礼抽奖系统

---

## 一、简历版本（精简版）

### 项目名称
**毕业典礼实时互动抽奖系统**

### 项目描述
为大学毕业典礼设计并开发的高并发实时互动系统，支持 2000+ 用户同时在线参与弹幕互动和实时抽奖。系统采用前后端分离架构，包含大屏展示端、移动端和管理后台三端应用，实现了实时弹幕推送、滚动抽奖动画、敏感词过滤、用户权限管理等核心功能。

### 技术栈
**后端**：Spring Boot 3.2.5 + MyBatis-Plus + Spring Security + JWT + WebSocket + Redis + MySQL 8.0

**前端**：Vue 3 + Vite + Composition API

**部署**：Docker + Docker Compose + Nginx + 阿里云 ECS

### 负责模块
1. **后端架构设计**：分层架构（Controller → Service → Mapper），RESTful API 设计
2. **实时通信模块**：WebSocket 实现弹幕推送、抽奖状态同步、在线人数统计
3. **安全认证体系**：Spring Security + JWT 实现管理后台无状态认证
4. **高并发优化**：Redis 缓存敏感词、分布式在线用户管理、乐观锁防止重复中奖
5. **三端前端开发**：大屏展示端（弹幕动画、抽奖滚动）、移动端（扫码注册、弹幕发送）、管理后台（活动管理、用户管理）
6. **容器化部署**：Docker 镜像构建、Docker Compose 编排、Nginx 反向代理

### 项目亮点
- 使用 WebSocket 实现 2000+ 用户实时弹幕推送，延迟 < 100ms
- Redis 缓存敏感词库，弹幕发送响应时间从 50ms 降至 < 5ms
- 乐观锁 + 数据库事务保证抽奖并发安全，避免重复中奖
- JWT 无状态认证，支持分布式部署
- Docker 一键部署，包含 MySQL、Redis、后端、前端四个容器

## 二、STAR 法则版本

### Situation（背景）
大学毕业典礼需要一个实时互动系统，支持 2000+ 名毕业生同时参与弹幕互动和在线抽奖。系统需要在大屏幕上实时展示弹幕和抽奖动画，学生通过手机扫码参与，管理员通过后台控制活动流程。

### Task（任务）
作为独立开发者，负责整个系统的设计、开发和部署：
1. 设计后端架构，实现活动管理、弹幕系统、抽奖系统等核心功能
2. 开发三端前端应用，实现不同场景的用户交互
3. 保证系统在高并发场景下的稳定性和实时性
4. 实现安全的管理后台认证体系
5. 完成容器化部署，确保生产环境稳定运行

### Action（行动）

**1. 架构设计**
- 采用前后端分离架构，后端 Spring Boot + 前端 Vue 3
- 设计 RESTful API，使用 Swagger 自动生成接口文档
- 分层架构：Controller → Service → Mapper，职责清晰

**2. 实时通信实现**
- 使用 WebSocket 实现全双工通信
- 设计消息类型：弹幕（danmaku）、抽奖开始（lottery_start）、抽奖结果（lottery_result）、在线人数（online_update）
- 使用 ConcurrentHashMap 管理连接，支持一对多映射（一个用户多个标签页）

**3. 高并发优化**
- Redis 缓存敏感词库，避免每次发弹幕都查询数据库
- Redis Set 存储在线用户 ID，支持分布式部署
- 数据库乐观锁（@Version）防止并发抽奖导致重复中奖
- 异步广播消息，避免阻塞主线程

**4. 安全认证**
- Spring Security + JWT 实现无状态认证
- 公开接口（活动查询、弹幕发送）无需认证
- 管理接口（活动管理、用户管理）需要 JWT Token
- BCrypt 密码加密存储

**5. 前端开发**
- 大屏端：Canvas 绘制二维码、CSS 动画实现弹幕滚动、抽奖滚动动画
- 移动端：响应式设计、学号快速登录、WebSocket 实时接收弹幕
- 管理端：活动 CRUD、抽奖轮次管理、用户禁言/移出、敏感词管理

**6. 部署优化**
- Docker 多阶段构建，减小镜像体积
- Nginx 配置 Gzip 压缩，减少传输体积 60%-70%
- 静态资源 7 天缓存，减少重复请求
- Docker Compose 编排四个服务：MySQL、Redis、后端、前端

### Result（结果）
1. **性能指标**
   - 弹幕延迟 < 100ms，支持 2000+ 用户同时在线
   - 敏感词检查响应时间从 50ms 降至 < 5ms（Redis 缓存）
   - 页面加载时间 < 2s（Gzip 压缩 + 静态资源缓存）

2. **功能完成度**
   - 完整的三端应用：大屏端、移动端、管理后台
   - 核心功能：实时弹幕、滚动抽奖、敏感词过滤、用户管理
   - 安全认证：JWT 无状态认证、权限控制

3. **部署效果**
   - Docker 一键部署，包含 4 个容器
   - 阿里云 ECS 生产环境稳定运行
   - 支持局域网和公网访问

## 四、可以优化的方向（建议实现）

### 🔴 高优先级（简历加分项）

#### 1. 高并发深度优化
**现状**：使用 Redis 缓存 + 乐观锁
**可优化**：
- Redis 分布式锁（Redisson）控制抽奖并发
- 消息队列（RabbitMQ/Kafka）异步处理弹幕广播
- 限流（Sentinel）防止恶意请求
- 连接池优化（Redis Lettuce Pool、MySQL HikariCP）

**简历话术**：
> "使用 Redisson 分布式锁保证抽奖并发安全，结合 RabbitMQ 异步处理弹幕广播，支持 5000+ 用户同时在线"

#### 2. 安全体系完善
**现状**：Spring Security + JWT
**可优化**：
- 接口限流（防刷）
- XSS 防护
- CSRF 防护
- SQL 注入防护（MyBatis-Plus 已支持）
- 敏感数据加密（手机号等）
- 操作日志审计

**简历话术**：
> "实现完整的安全防护体系，包括 JWT 认证、接口限流、XSS/CSRF 防护、敏感数据加密、操作日志审计"

#### 3. 高可用架构
**现状**：单机部署
**可优化**：
- Nginx 负载均衡（多后端实例）
- MySQL 主从复制
- Redis Sentinel 哨兵模式
- 服务降级（Sentinel）
- 健康检查（Spring Boot Actuator）

**简历话术**：
> "设计高可用架构，Nginx 负载均衡 + MySQL 主从 + Redis Sentinel，保证服务 99.9% 可用性"

### 🟡 中优先级（技术深度）

#### 4. 数据一致性
**现状**：@Transactional + 乐观锁
**可优化**：
- 分布式事务（Seata）- 如果有跨服务操作
- 数据库索引优化
- 慢 SQL 监控
- 数据库连接池调优

#### 5. 实时通知增强
**现状**：WebSocket 基础实现
**可优化**：
- 心跳检测（Ping/Pong）
- 断线重连优化
- 消息确认机制
- 离线消息存储

#### 6. 监控与运维
**现状**：基础日志
**可优化**：
- Spring Boot Actuator 健康检查
- Prometheus + Grafana 监控
- ELK 日志收集
- 链路追踪（SkyWalking）

### 🟢 低优先级（锦上添花）

#### 7. 开发效率
- 单元测试（JUnit 5 + Mockito）
- 集成测试
- CI/CD（GitHub Actions）
- 代码质量检查（SonarQube）

#### 8. 微服务化（如果项目复杂度提升）
- Spring Cloud Gateway
