# 20260519150000-三个Bug修复

## 问题提示词

用户反馈了三个问题：
1. 在后台的任意一个tab中，点击刷新都会跳到活动管理的tab
2. 将一个用户移除直播间后，他就不能以原来的学号再登陆进来了
3. 当没有可供选择的参与者的时候，screen页面应该没有任何反应，而不是数字一直循环播放
4. 历史活动、屏幕设置、用户管理三个 tab 刷新后页面数据为空（弹幕审核也有同样问题）

## 做了什么事情

### 1. Admin Tab 持久化

**修改文件**：`lucky-admin/src/App.vue`

**修改内容**：
- 第 363 行：`currentTab` 初始化时从 `localStorage` 读取已保存的 tab
  ```js
  const currentTab = ref(localStorage.getItem('admin_currentTab') || 'activity')
  ```
- 第 1025 行 watch 中：每次切换 tab 时写入 `localStorage`
  ```js
  localStorage.setItem('admin_currentTab', tab)
  ```

### 2. 被移出用户禁止重新登录

**修改文件**：`lucky-server/.../service/impl/ParticipantServiceImpl.java`

**修改内容**：在 `checkRegistered()` 方法中，查到参与者后检查 `is_banned` 字段。如果用户已被移出直播间（`is_banned = 1`），抛出异常阻止登录：
```java
if (p != null && p.getIsBanned() != null && p.getIsBanned() == 1) {
    throw new RuntimeException("您已被移出直播间，无法进入");
}
```

Mobile 端 `quickEnter()` 已有 `data.code !== 200` 时显示错误信息的逻辑，无需修改前端。

### 3. 修复无参与者时抽奖动画时序

**修改文件**：
- `lucky-server/.../controller/LotteryController.java`
- `lucky-server/.../service/impl/LotteryServiceImpl.java`

**修改内容**：

**LotteryController**：删除了 `draw()` 方法中在调用 service 之前广播 `lottery_start` 的代码，同时移除了不再使用的 `webSocketHandler` 字段和 import。

**LotteryServiceImpl**：在 `draw()` 方法中，将 `lottery_start` 广播移到参与者检查通过之后、实际抽奖之前：
```java
if (candidates.isEmpty()) {
    throw new RuntimeException("没有可抽奖的参与者");
}

// 通知大屏开始抽奖动画（参与者检查通过后再广播）
webSocketHandler.broadcast("lottery_start", roundId);

// 随机打乱
Collections.shuffle(candidates);
```

### 4. Tab 刷新后数据不加载

**修改文件**：`lucky-admin/src/App.vue`

**修改内容**：在 `onMounted` 中，等 `fetchCurrentActivity()` 完成后再根据当前 `currentTab` 加载对应数据。因为 `fetchAllUsers()`、`fetchOnlineUsers()`、`fetchPendingDanmakus()` 都依赖 `currentActivity.value`（有 `if (!currentActivity.value) return` 前置检查），必须等活动数据加载完成：
```js
onMounted(() => {
  fetchCurrentActivity().then(() => {
    // 加载当前 tab 的数据（刷新后 watch 不触发，需手动加载）
    // 部分 tab 依赖 currentActivity，需等其加载完成
    const tab = currentTab.value
    if (tab === 'danmaku') fetchPendingDanmakus()
    if (tab === 'screen') fetchScreenConfig()
    if (tab === 'history') fetchHistoryActivities()
    if (tab === 'users') {
      fetchAllUsers()
      fetchOnlineUsers()
    }
  })
  fetchSensitiveWords()
})
```

## 为什么这样做

### Bug 1：Tab 持久化
原实现中 `currentTab` 默认值为 `'activity'`，浏览器刷新时 SPA 重新加载导致状态丢失。通过 localStorage 持久化，刷新后能恢复到用户所在的 tab。

### Bug 2：服务端校验
原 `checkRegistered()` 只按 `activityId` + `studentId` 查询，不检查 `is_banned` 字段，导致被移出的用户仍可通过快速进入登录。在服务端添加校验是最安全的做法，不依赖前端判断。

### Bug 3：广播时序
原实现中 `LotteryController.draw()` 在调用 `lotteryService.draw()` **之前**就广播了 `lottery_start`。当 service 发现没有参与者抛出异常时，大屏已经收到 `lottery_start` 并开始播放占位符名称的滚动动画。将广播移到 service 中参与者检查通过之后，确保只有真正能抽奖时才触发动画。

### Bug 4：Tab 刷新后数据不加载
`watch(currentTab, ...)` 只在值**变化**时触发。浏览器刷新后 `currentTab` 从 localStorage 恢复为非默认值（如 `'users'`），这是初始值而非变化，watch 不触发。`onMounted` 中只调了 `fetchCurrentActivity()` 和 `fetchSensitiveWords()`，导致 `danmaku`、`screen`、`history`、`users` 四个 tab 的数据不加载。此外，`fetchAllUsers()`、`fetchOnlineUsers()`、`fetchPendingDanmakus()` 都有 `if (!currentActivity.value) return` 的前置检查，`fetchCurrentActivity()` 是异步函数，如果不同步等待其完成，这些函数会因 `currentActivity` 为 null 而直接返回。通过 `.then()` 链式调用确保活动数据加载完成后再加载当前 tab 数据。

## 产生的结果

- Admin 后台刷新浏览器后，停留在当前 tab 而非跳回"活动管理"
- 被移出直播间的用户无法再用原学号快速进入，会收到"您已被移出直播间，无法进入"错误提示
- 没有可抽奖参与者时，点击抽奖按钮后大屏无动画反应，admin 端显示"没有可抽奖的参与者"错误提示
- 刷新浏览器后，当前 tab 的数据正常加载（历史活动、屏幕设置、用户管理、弹幕审核均已验证）
- 三个项目（lucky-server、lucky-admin、lucky-mobile）构建均通过，无报错
