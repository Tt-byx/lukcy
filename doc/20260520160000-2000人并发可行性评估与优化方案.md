## 三、2000人并发瓶颈分析

### 3.1 瓶颈一：服务器硬件不足（致命）

| 资源 | 现状 | 2000并发需求 | 差距 |
|------|------|-------------|------|
| 内存 | 2GB | 至少 4-8GB | **严重不足** |
| 带宽 | 阿里云学生机通常 1Mbps | 至少 5-10Mbps | **严重不足** |
| CPU | 2核 | 至少 4核 | 不足 |

**内存计算：**
- JVM 堆内存（Spring Boot）：至少需要 1GB+
- MySQL：至少需要 500MB+
- 2000个 WebSocket 连接：每个约 50KB-200KB，共约 100MB-400MB
- 操作系统：约 500MB
- **总计：约 2.1GB-2.9GB，远超 2GB 可用**

**带宽计算：**
- 1Mbps ≈ 128KB/s
- 2000人平分：每人仅 64B/s
- 弹幕消息一条 JSON 约 200 字节，2000人同时收到一条弹幕需要传输 400KB
- 400KB ÷ 128KB/s ≈ **3秒才能发完一条弹幕**

### 3.2 瓶颈二：WebSocket broadcast 严重阻塞（致命）

```java
// LuckyWebSocketHandler.java - 当前实现
public void broadcast(String type, Object data) {
    // ...构造消息...
    for (WebSocketSession session : sessionMap.values()) {
        if (session.isOpen()) {
            synchronized (session) {          // ❌ 每个session加锁
                try {
                    session.sendMessage(textMessage);  // ❌ 串行发送
                } catch (IOException | IllegalStateException e) {
                    // ...
                }
            }
        }
    }
}
```

**问题分析：**
- 2000个连接，每次broadcast需要串行遍历2000次
- 每个 `sendMessage` 是同步阻塞IO操作
- `synchronized(session)` 虽然保护了单个session的并发写入，但2000个串行发送会导致整条链路阻塞
- 一条弹幕的broadcast耗时 = 2000 × 单次发送耗时 ≈ 2000 × 1ms ≈ **2秒**
- 如果2000人同时发弹幕，后面的人要等很久才能看到

### 3.3 瓶颈三：数据库连接池耗尽

- HikariCP 默认最大连接数：**10**
- 2000人并发请求，即使只有20%的人在操作数据库，也有400个并发请求
- 10个连接完全不够，大量请求会排队等待，最终超时

### 3.4 瓶颈四：Tomcat 线程池不足

- Spring Boot 内嵌 Tomcat 默认最大线程数：**200**
- 每个HTTP请求占用一个线程
- 2000人并发，线程池瞬间打满，新请求全部排队/拒绝

### 3.5 瓶颈五：弹幕发送链路过长

```
用户点击发送
  → HTTP POST /api/danmaku/send
    → 敏感词检查（数据库查询）
    → INSERT INTO danmaku（数据库写入）
    → broadcast() 遍历所有session（串行IO）
  → 返回响应给用户
```

整个链路：**数据库查询 → 数据库写入 → 2000次串行IO → 返回响应**

用户从点击发送到看到自己的弹幕出现在屏幕上，可能需要2-3秒甚至更长。

### 3.6 瓶颈六：视频/背景资源

- 大屏播放视频背景，每个mobile端也可能有视频背景
- 资源托管在阿里云OSS，不受服务器带宽影响（这个还好）
- 但mobile端的静态资源（JS/CSS）需要从服务器加载，2000人同时首次加载会产生瞬时流量峰值

---

## 四、结论：当前架构完全无法支撑2000并发

| 瓶颈点 | 严重程度 | 影响 |
|--------|---------|------|
| 2G内存 | **致命** | 机器直接OOM崩溃 |
| 1Mbps带宽 | **致命** | 弹幕延迟3秒+，页面加载极慢 |
| WebSocket串行广播 | **致命** | 弹幕发送卡死 |
| 数据库连接池10 | **严重** | 大量请求超时 |
| Tomcat线程池200 | **严重** | HTTP请求排队积压 |
| 无缓存中间件 | **中等** | 每次请求都查数据库 |

**结论：2G学生机在无任何中间件的情况下，最多支撑 50-100 人同时在线。2000人绝对无法支撑。**

## 五、优化方案

### 5.1 必须添加的中间件和配置

#### 方案一：最小改动方案（支撑到 500 人）

| 组件 | 用途 | 重要性 |
|------|------|--------|
| **Nginx** | 反向代理 + 静态资源 + 限流 + gzip | ★★★★★ |
| **Redis** | 在线人数缓存 + 活动状态缓存 + 弹幕队列 | ★★★★★ |
| 升级带宽 | 至少 3-5Mbps | ★★★★★ |

#### 方案二：推荐方案（支撑 2000 人）

| 组件 | 用途 | 重要性 |
|------|------|--------|
| **Nginx** | 反向代理、静态资源、负载均衡、限流、gzip压缩 | ★★★★★ |
| **Redis** | 在线人数、活动状态、参与者缓存、弹幕消息队列（Pub/Sub） | ★★★★★ |
| **升级服务器** | 至少 4核8G + 5Mbps带宽 | ★★★★★ |
| **消息队列**（可选） | RocketMQ/RabbitMQ 异步处理弹幕 | ★★★★ |

### 5.2 具体改造步骤

#### Step 1：服务器升级（必须）

```
建议配置：
- CPU：4核
- 内存：8GB
- 带宽：5Mbps（按量付费或固定带宽）
- 系统盘：40GB（够用）
```

#### Step 2：安装 Nginx

```bash
# CentOS
yum install -y nginx
# Ubuntu
apt install -y nginx
```

**Nginx 配置示例：**

```nginx
upstream lucky_backend {
    server 127.0.0.1:8080 weight=1;
}

server {
    listen 80;
    server_name your-domain.com;

    # gzip 压缩（减少带宽消耗70%+）
    gzip on;
    gzip_min_length 1k;
    gzip_comp_level 6;
    gzip_types text/plain text/css text/javascript application/json application/javascript;

    # 静态资源（前端dist文件）
    location /screen/ {
        alias /opt/lucky/lucky-screen/dist/;
        try_files $uri $uri/ /screen/index.html;
    }
    location /mobile/ {
        alias /opt/lucky/lucky-mobile/dist/;
        try_files $uri $uri/ /mobile/index.html;
    }
    location /admin/ {
        alias /opt/lucky/lucky-admin/dist/;
        try_files $uri $uri/ /admin/index.html;
    }

    # API 反向代理
    location /api/ {
        proxy_pass http://lucky_backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }

    # WebSocket 反向代理
    location /ws/ {
        proxy_pass http://lucky_backend;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_read_timeout 86400s;
    }

    # 限流：每个IP每秒最多10个请求
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;
    location /api/ {
        limit_req zone=api_limit burst=20 nodelay;
        proxy_pass http://lucky_backend;
    }
}
```

**改造1：在线人数用 Redis 计数（替代 WebSocket broadcast）**

```java
// 使用 Redis 的 SET 存储在线用户
@Resource
private RedisTemplate<String, Object> redisTemplate;

private static final String ONLINE_KEY = "lucky:online:participants";

// 上线时
redisTemplate.opsForSet().add(ONLINE_KEY, participantId.toString());

// 下线时
redisTemplate.opsForSet().remove(ONLINE_KEY, participantId.toString());

// 获取在线人数
Long onlineCount = redisTemplate.opsForSet().size(ONLINE_KEY);

// 定时（每5秒）通过WebSocket广播在线人数，而不是每次连接/断连都广播
@Scheduled(fixedDelay = 5000)
public void syncOnlineCount() {
    Long count = redisTemplate.opsForSet().size(ONLINE_KEY);
    broadcast("online_update", count.intValue());
}
```

**改造6：WebSocket 并发发送优化**

```java
// 使用线程池并发发送，替代串行遍历
private final ExecutorService broadcastExecutor = 
    Executors.newFixedThreadPool(20);

public void broadcast(String type, Object data) {
    JSONObject msg = new JSONObject();
    msg.put("type", type);
    msg.put("data", data);
    String json = msg.toJSONString();
    TextMessage textMessage = new TextMessage(json);

    List<WebSocketSession> sessions = new ArrayList<>(sessionMap.values());
    for (WebSocketSession session : sessions) {
        broadcastExecutor.submit(() -> {
            if (session.isOpen()) {
                synchronized (session) {
                    try {
                        session.sendMessage(textMessage);
                    } catch (Exception e) {
                        log.error("发送失败: {}", session.getId(), e);
                    }
                }
            }
        });
    }
}
```

## 八、总结

1. **当前 2G 学生机无法支撑 2000 人**，必须升级服务器到至少 4核8G + 5M 带宽。
2. **必须添加 Nginx**：用于静态资源服务、gzip压缩、反向代理、限流。
3. **必须添加 Redis**：用于在线人数缓存、弹幕消息队列、活动状态缓存。
4. **必须改造 WebSocket broadcast**：串行改并发线程池发送。
5. **必须调优数据库连接池和Tomcat线程池**。

如果预算极其有限，建议：
- 先升级带宽到 3Mbps（临时按量付费）
- 安装 Nginx + Redis
- 进行代码改造
- 预计可支撑 300-500 人

2000 人规模的年会抽奖，服务器成本是绕不过去的门槛，建议提前申请预算升级服务器。
